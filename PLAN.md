# 多跳检索 Agent 升级方案

> 把现有「一问一答的检索链」升级为「会自己判断信息够不够、还要查什么」的多跳 Agent。
> 核心思路:**复用现有检索栈,不重写**;只在外层加规划/反思/跳数控制。

---

## 一、为什么做 / 做完什么样

**现状**:一次 query → 一次混合检索(向量 + BM25 + RRF + rerank) → 一次生成。
遇到「被辞退没补偿能不能告、怎么告」这类问题,单次检索召回的条文不完整,LLM 只能基于残缺上下文硬答。

**目标**:系统能自己判断「信息够不够、还要查什么」,多轮检索后再作答。

**验收标准(先定好,才知道做没做成)**:
- 准备一组 ~20 道**多跳测试题**(答案需要 ≥2 次不同检索才能完整)。
- 对比 Agent 版 vs 现有单跳版,在这组题上的:
  - **答案完整性**(复用现有 LLM-as-judge 打分)
  - **引用覆盖率**(命中的相关法条比例)
- 守住底线:单跳就能答的简单题,Agent 版**不能明显变慢、不能退化**(回归测试)。

---

## 二、目标架构(LangGraph 状态机)

```
                  ┌─────────────┐
   用户提问 ───►   │  Planner    │  判断:能直接答? 还是要拆成子问题?
                  └──────┬──────┘
                         │ 子问题队列
              ┌──────────▼───────────┐
              │   Retrieve            │  ◄── 复用 vector_store + bm25 + rrf + reranker
              │   (现有混合检索 tool)  │
              └──────────┬───────────┘
                         │ 召回条文
                  ┌──────▼──────┐
                  │  Reflect    │  够了吗? 还缺什么?(最多 N 跳,防死循环)
                  └──┬───────┬──┘
              不够   │       │  够了
            (回 Planner)     │
                         ┌───▼────┐
                         │ Answer │  复用现有 prompt + 引用校验
                         └────────┘
```

核心是 **Reflect 节点** + **跳数上限**(如 `max_hops=3`)——这是 Agent 与死循环之间的唯一区别。

## 三、具体改造点(基于已有文件)

| 模块 | 现状 | 改造 |
|---|---|---|
| `src/vector_store.py` | 混合检索函数 | **几乎不动**,包一层 `retrieve_tool(query) -> List[Article]` |
| `src/reranker.py` / `src/bm25_retriever.py` | 已有 | 不动,被 retrieve_tool 内部使用 |
| `src/rag.py` | 端到端单跳链 | 保留作为「简单模式」;多跳逻辑放到新文件 |
| **新增 `src/agent.py`** | — | LangGraph 图:Planner / Retrieve / Reflect / Answer 四节点 + 状态(已查问题、已得条文、跳数) |
| **新增 `src/tools.py`** | — | 检索、法条交叉引用(顺「依照第X条」拉关联)、时效计算等包装成 tool |
| `src/web.py` | Gradio 界面 | 加「深度模式」开关 + 展示 Agent 检索轨迹(查了几次) |
| `eval/` | 单跳评估集 | 新增 `multihop_set.json` + 评估脚本 |
| `src/config.py` | 配置 | 加 `MAX_HOPS`、`AGENT_MODE` 等参数 |

**关键设计取舍**:
- **模型分级**:Planner / Reflect 用便宜快的模型,Answer 用强模型——压住多跳放大的 API 成本(顺势复用已有的异构故障转移)。
- **复用而非重写**:检索栈一行不改,只是被「调用」,改造风险低。
- **可降级**:Planner 判断是简单问题就直接走老的单跳 `rag.py`,不为难简单题。

## 四、分期实施(每期都能独立验证)

### Phase 0 — 基线(~0.5 天)
造 20 道多跳测试题,先跑现有单跳系统并记录得分。
> ⚠️ 没有这个基线,后面无法证明 Agent 更好。

### Phase 1 — 检索工具化(~1 天)
把混合检索包成 tool,手写一个最简单的「固定两跳」循环(先不用 LangGraph),验证多跳确实召回更全。

### Phase 2 — LangGraph 化(~2-3 天)【核心】
引入 LangGraph,实现 Planner / Retrieve / Reflect / Answer 四节点 + 跳数上限。

### Phase 3 — 工具扩展(~1-2 天,可选)
加「法条交叉引用」「时效计算」等专用 tool,让 Agent 会用工具而不只是反复检索。

### Phase 4 — 界面 + 评估(~1 天)
Gradio 加深度模式开关 + 轨迹展示;补多跳评估脚本,出 Agent vs 单跳对比报告。

**总计约 5-7 个工作日。Phase 0-2 是最小可用版本(MVP),做完即可展示。**

## 五、风险与注意点

- **死循环 / 成本爆炸**:必须有 `max_hops` 硬上限 + 每跳去重(别反复查同一子问题)。
- **Agent 不一定更好**:简单题上 Agent 可能画蛇添足 → 保留单跳模式 + 回归测试是硬要求。
- **延迟**:多跳会明显变慢(2-3 倍)→ 深度模式做成可选,简单问题走快路径。
- **可观测性**:多跳调试很痛苦 → Phase 2 起就打 trace 日志(每跳查了啥、Reflect 怎么判断),否则定位不了问题。

---

## 六、技术栈补充

- **编排框架**:LangGraph(状态机式,可控、可调试,比 LangChain 黑盒 Agent 好排错)
- **Agent 范式**:Plan-Execute + Reflect(也可手写 ReAct 循环理解原理)
- **LLM**:沿用现有 ModelScope API(DeepSeek-V3.2 / Qwen3 / GLM-5),按节点分级调用
- **Trace**:LangSmith / Phoenix / 自建日志(Phase 2 起接入)
